Section 1

Assessment Overview & Project Setup

What is dbt?

dbt (data build tool) is an open-source transformation framework that allows data engineers and analysts to write modular SQL and Jinja-based transformations that are version-controlled, tested, and documented. dbt handles the T in ELT — it transforms data that has already been loaded into a data warehouse.

Key capabilities used in this assessment: materialized models (tables/views/incremental), snapshots for SCD Type 2, built-in data testing, macros from the dbt_utils package, and selectors for targeted execution.

Tech Stack

Database

DuckDB — an in-process analytical database. AdventureWorks data is stored in adventureworks.duckdb. Schemas: production (source data) and mart (transformed output).

dbt Adapter

dbt-duckdb 1.7.3 with packages dbt_utils 1.3.1 (surrogate keys, macros) and codegen 0.13.1 (YAML scaffolding).

Schema Routing

Custom generate_schema_name macro — uses the custom schema name directly instead of the default target_schema__custom_schema pattern. Snapshots go to mart schema.

Data Loading

16 CSV seed files in seeds/production/ loaded via dbt seed. Includes product, sales orders, customers, geography tables. Mart seed: happiest_place_on_earth.

Task 1

SCD Type-2 Snapshot — dim_product_scd2

File: snapshots/dim_product_scd2.sql  |  Target schema: mart

What is a Slowly Changing Dimension Type 2?

A Slowly Changing Dimension Type 2 (SCD2) is a data warehousing pattern that preserves the full history of changes to a dimension. Instead of overwriting the old value, it inserts a new row and flags the old row as expired. This allows analysts to answer questions like "What was the product's price when this order was placed?" — a question SCD Type 1 cannot answer.

SCD2 Adds These dbt Columns Automatically

  • ✓
    dbt_scd_id — hash of unique_key + check_cols, identifies each version
  • ✓
    dbt_updated_at — when dbt processed this row
  • ✓
    dbt_valid_from — when this version became active
  • ✓
    dbt_valid_to — when this version expired (NULL = currently active)

Check Strategy vs Timestamp Strategy

dbt offers two snapshot strategies:

  • ✓
    check — hashes specified columns and detects changes by value comparison. Used here.
  • i
    timestamp — compares an updated_at column. Faster but requires a reliable timestamp on the source.

We used check because date_modified in the product seed isn't reliably updated on every attribute change.

Requirements Fulfilled

  • ✓
    Req 1: SCD Type 2 implemented via dbt {% snapshot %} block — dbt manages inserts of new versions and expiry of old ones automatically.
  • ✓
    Req 2: Surrogate key generated using dbt_utils.generate_surrogate_key(['p.product_id']) — an MD5 hash of product_id, giving a stable, non-sequential key.
  • ✓
    Req 3: strategy='check' configured — detects changes by comparing hashed values of the monitored columns.
  • ✓
    Req 4: All 6 change-detection columns included in check_cols: product_name, product_number, color, list_price, product_category_name, product_subcategory_name.
  • ✓
    Req 5: Written to mart schema via target_schema='mart'. History verified: updating a product row and re-running dbt snapshot inserts a new version row with the old row's dbt_valid_to stamped.

Key Design Decisions

Why LEFT JOIN instead of INNER JOIN?

504 products are seeded, but many have product_subcategory_id = NULL (raw materials, sub-assembly parts). An INNER JOIN would silently drop these rows from the dimension — a data loss that would corrupt any fact-to-dimension join. LEFT JOIN retains all products and populates category columns as NULL for uncategorized products.

Why product_id as unique_key (not product_key)?

The unique_key in a snapshot is dbt's anchor — it identifies which entity changed so dbt can expire the old version and insert the new one. Using the surrogate key here would break this because the surrogate key is derived from product_id — they'd always match. product_id is the natural business key that stays constant across versions.

How to Verify History Tracking 1. Run dbt seed + dbt snapshot (initial load — all rows have dbt_valid_to = NULL).
2. Edit a row in the product CSV (e.g. change list_price for product_id=1).
3. Run dbt seed --full-refresh then dbt snapshot again.
4. Query: SELECT * FROM mart.dim_product_scd2 WHERE product_id = 1 ORDER BY dbt_valid_from;
— You will see 2 rows: old version with dbt_valid_to stamped, new version with dbt_valid_to = NULL.

Task 2

SCD Type-1 Incremental Model — dim_product_scd1

File: models/marts/dim_product_scd1.sql  |  Materialization: incremental

SCD Type 1 vs SCD Type 2

SCD Type 1 (This Task)

  • ✓
    Overwrites old values — no history kept
  • ✓
    Always reflects the current state of the source
  • ✓
    Use when history is irrelevant (e.g. correcting a typo)
  • ✓
    Simpler queries — no need to filter for current version

SCD Type 2 (Task 1)

  • ✓
    Inserts new rows on change — full history preserved
  • ✓
    Supports "as-of" reporting ("what was the price then?")
  • ✓
    More complex queries — must filter WHERE dbt_valid_to IS NULL
  • ✓
    Table grows over time with each change

How Incremental Materialization Works

On the first run, dbt creates the table by running the full SELECT. On subsequent runs, the {% if is_incremental() %} block activates and filters the source to only new or changed rows. dbt then MERGEs these into the existing table using the unique_key to upsert (update existing + insert new).

Watermark Logic WHERE p.date_modified > (SELECT MAX(date_modified) FROM this) — {{ this }} is a dbt Jinja reference to the model's own table in the warehouse. This pattern reads the highest timestamp already loaded and only processes rows newer than that boundary.

  • ✓

  • ✓
    Req 8: Incremental materialization with date_modified watermark filter.
  • ✓
    Req 9: Updating a product and reseeding, then running dbt run, propagates the change into dim_product_scd1 — old value is overwritten.
  • ✓

Delete Behavior — Written Answer (Requirement 10)

What Happens When a Row is Deleted from the Source?

The deleted product row remains in dim_product_scd1. The incremental watermark only compares date_modified timestamps — a deleted row produces no new timestamp event, so dbt never processes it. The stale row silently persists.

What SHOULD Happen (SCD Type 1 Principle)?

Since SCD1 represents current state only, deleted source rows should be removed from the dimension. Three viable solutions:

  1. Full refresh: dbt run --full-refresh rebuilds the table from scratch — simplest but most expensive at scale.
  2. delete+insert strategy: Set incremental_strategy='delete+insert' with a predicate that covers all product IDs in scope — dbt deletes the old partition then re-inserts.
  3. Post-hook DELETE: Add a post-hook that deletes from the target any product_key values not present in the current source query.
Task 3

Data Tests

File: models/marts/schema.yml  |  Tests run: dbt test

dbt Testing Architecture

Generic Tests (Used Here)

Defined in schema.yml files inside model directories. dbt's four built-in generic tests: not_null, unique, accepted_values, relationships. Applied by column in YAML — no SQL needed.

Singular Tests

Custom .sql files in the tests/ directory. A test passes if the query returns zero rows. Used for complex business logic that generic tests can't express.

Why tests/ directory vs models/ directory? dbt's test-paths (the tests/ folder) only loads .sql singular tests. Generic test YAML must live in a model path (model-paths). Placing schema.yml in tests/ would make dbt ignore it. The correct location for not_null/unique tests is models/marts/schema.yml.

Why These Tests Matter

product_key — not_null + unique

The surrogate key is the primary join key between the dimension and any fact table. If it contains NULLs, fact rows will fail to join. If it has duplicates, joins will fan out and inflate metrics (a silent, dangerous bug). These two tests together enforce primary key integrity.

product_id — not_null

The natural business key must always be present. A NULL product_id would mean the surrogate key hash is computed on NULL input — producing identical surrogate keys for multiple products and breaking uniqueness.

Implementation

version: 2

models:
  - name: dim_product_scd1
    columns:
      - name: product_key
        tests:
          - not_null
          - unique
      - name: product_id
        tests:
          - not_null

Task 4

Tags & Selectors

Files: selectors.yml & tag in dim_product_scd1.sql

Why Tags and Selectors?

In production dbt projects with hundreds of models, running all models for every pipeline is wasteful and slow. Tags and selectors let you target specific subsets of the DAG — by domain, by schedule, by team ownership. A tag is metadata attached to a model; a selector is a named, reusable way to select models by that metadata (or other criteria).

Tag Added to Model Config

Tags can be set at model-level (in config()), directory-level (in dbt_project.yml), or in schema.yml. Here we tag inline in the config block for clarity:

config(
    materialized='incremental',
    unique_key='product_key',
    tags=['product']   ← here
)

selectors.yml

Selectors define named selection criteria. Multiple criteria can be combined with union or intersection:

selectors:
  - name: product_models
    description: All models tagged 'product'
    definition:
      method: tag
      value: product

Running With the Selector

# Run only models tagged 'product'
dbt run --selector product_models

# Output:
# 1 of 1 START sql incremental model marts.dim_product_scd1 ... [RUN]
# 1 of 1 OK created sql incremental model marts.dim_product_scd1 ... [OK in 0.93s]
# Done. PASS=1 WARN=0 ERROR=0 SKIP=0 TOTAL=1
Selectors vs --select flag dbt run --select tag:product achieves the same result inline, but selectors.yml is reusable, version-controlled, and can encode complex multi-criteria logic. Selectors also support union (OR) and intersection (AND) of criteria — e.g. "all models tagged product AND in the marts directory."
Task 5

3-Month Rolling Average SQL

File: task5_solution.sql  |  Run via: python query.py -f task5_solution.sql

Business Question

Calculate the 3-month rolling average of total monthly sales. The average must return NULL for the first two months because there is not yet enough data to compute a full 3-month window. From month 3 onward, the average covers the current month plus the two preceding months.

Query Breakdown

Step 1 — Aggregate to Monthly Sales (CTE)

with monthly_sales as (
    select
        date_trunc('month', h.order_date)::date as order_month,
        sum(d.order_qty * d.unit_price)         as total_monthly_sales
    from production.sales_order_detail as d
    join production.sales_order_header as h
        on d.sales_order_id = h.sales_order_id
    group by date_trunc('month', h.order_date)::date
)

date_trunc('month', ...) floors any date to the first of its month (e.g. 2011-07-15 → 2011-07-01). The ::date cast removes the time component. order_qty * unit_price computes line-level revenue, then SUM aggregates across all products in the month.

Step 2 — Window Function with NULL Enforcement

select
    order_month,
    total_monthly_sales,
    case
        when count(*) over (
            order by order_month
            rows between 2 preceding and current row
        ) = 3
        then avg(total_monthly_sales) over (
            order by order_month
            rows between 2 preceding and current row
        )
        else null
    end as three_month_rolling_avg
from monthly_sales
order by order_month

Window Function Mechanics

ClauseWhat it does
ORDER BY order_monthDefines the sort order within the window — months are processed chronologically
ROWS BETWEEN 2 PRECEDING AND CURRENT ROWPhysical frame: exactly the current row + 2 rows before it in sort order
COUNT(*) OVER (...) = 3Counts how many rows are actually in the window frame — evaluates to 1 or 2 for the first rows
AVG(...) OVER (...)Computes the average only when the count confirms a full 3-row window is available
ROWS vs RANGE — Critical Distinction ROWS BETWEEN 2 PRECEDING AND CURRENT ROW counts physical rows — exactly the 3 rows in position. RANGE BETWEEN INTERVAL '2 months' PRECEDING AND CURRENT ROW counts by value distance — would include all rows within 2 months of the current date value, potentially including duplicate months or skipping gaps. For this use case with one row per month, ROWS is correct and more predictable.

Section 8

Suggested Improvements

The assessment requirements are fully met. The following improvements would make this production-grade:

🔒

Add invalidate_hard_deletes=true to the SCD2 Snapshot

Currently, a product deleted from the source stays in dim_product_scd2 as an open-ended record (dbt_valid_to = NULL forever). Adding invalidate_hard_deletes=true makes dbt automatically set dbt_valid_to when a source row disappears, properly closing out deleted records.

📋

Replace ref('product') with source() + a sources.yml

Seeds are a dev convenience but in production the data would come from a raw schema, not CSVs. Defining sources.yml enables source freshness checks (dbt source freshness), better lineage documentation, and easier migration to a real warehouse source without changing model SQL.

🧪

Add Snapshot-Level Data Tests

Create snapshots/schema.yml to add not_null on dbt_valid_from, unique on dbt_scd_id, and a singular test verifying no product has more than one row with dbt_valid_to IS NULL. These catch snapshot corruption early.

⚡

Use delete+insert Incremental Strategy for SCD1 Delete Handling

Configure incremental_strategy='delete+insert' with partition_by on a date column. This resolves the delete-blindness problem without requiring a full-refresh, and is significantly safer at scale than post-hook DELETEs.

🔢

Add on_schema_change='append_new_columns' to the Incremental Model

Currently, adding a new column to dim_product_scd1 without --full-refresh silently succeeds but the new column is missing from the table. Setting on_schema_change='append_new_columns' makes dbt automatically ALTER the table to add the new column, preventing silent data loss.

Add More Robust Data Tests

Extend schema.yml with: accepted_values on product_category_name (Bikes, Components, Clothing, Accessories), a relationships test linking product_id back to the seed, and a custom singular test asserting list_price >= 0. This turns the test suite from a safety net into a data contract.

Fix the Misplaced Snapshot Config in dbt_project.yml

The current dbt_project.yml has snapshots: +schema: marts nested under models: — dbt interprets this as a subdirectory config for a folder called "snapshots" inside the models path, not as snapshot configuration. Move it to a top-level snapshots: key to actually apply it.

Add CI/CD Pipeline Configuration

Add a .github/workflows/dbt_ci.yml (or equivalent) that runs dbt build --select state:modified+ on every PR using the --defer flag against the production manifest. This runs only changed models and their downstream dependents, keeping CI fast and cheap.

The rolling average query would be more maintainable as models/marts/fct_monthly_sales_rolling_avg.sql — it would be version-controlled, testable (e.g. not_null on order_month), and documented in the dbt DAG. The standalone task5_solution.sql file is outside the dbt graph entirely.
